## 3.1. Цель и границы задачи, архитектура решения, структура каталогов

*В этот раздел вошли бывшие пункты: **3.1** «Цель и границы», **3.2** «Общая архитектура», **3.10** «Структура каталогов».*

### Цель программного комплекса

Реализовать воспроизводимый конвейер: загрузка исторических котировок (и при необходимости ставок funding для USDT-M перпетуалов) → построение признаков без утечки информации из будущего → среда Gymnasium с явной моделью портфеля и издержками → обучение алгоритма **PPO** (Proximal Policy Optimization) из библиотеки **Stable-Baselines3** → оценка на отложенном отрезке → опционально вывод «сырого» сигнала веса позиции по последним свечам без отправки ордеров на биржу.

### Границы задачи

Реализация ориентирована на **исследовательский бэктест и демонстрацию инференса**, а не на промышленную торговую систему. Не моделируются очередь заявок, частичное исполнение, глубина стакана, режимы биржи, проскальзывание в явном виде (кроме упрощённой комиссии от оборота позиции). Это следует явно формулировать в тексте работы как допущения.

### Общая архитектура (логическая цепочка)

1. **Данные** хранятся в CSV: минимально `open`, `high`, `low`, `close`, `volume`; для склейки с Binance funding добавляется `open_time` (миллисекунды открытия свечи, UTC).
2. **Утилита `utils/data.py`** читает CSV, проверяет обязательные колонки, опционально извлекает ряд `funding_rate` / `funding` и строит матрицу признаков `X` и маску валидности `valid`.
3. **Среда `envs/crypto_env.py`** реализует один шаг симуляции: действие агента — целевая доля в активе `w`, наблюдение — склеенное окно признаков плюс нормированные признаки состояния портфеля.
4. **Скрипт `scripts/train_ppo.py`** разбивает ряд на train/eval по доле `train_ratio`, создаёт две обёртки среды, запускает PPO с колбэком сохранения лучшей модели по валидации и опционально пишет логи TensorBoard.
5. **Скрипт `scripts/eval_ppo.py`** загружает сохранённую политику и выводит в консоль метрики на выбранном сегменте (в т.ч. эталон buy-and-hold).
6. **Скрипт `scripts/live_signal.py`** строит наблюдение для последней свечи (из сети по публичному REST klines или из локального CSV) и печатает предложенный `w` без торговли.
7. Вспомогательные скрипты: **`make_sample_csv.py`** — синтетический ряд для отладки пайплайна; **`merge_binance_funding.py`** — подтягивание истории funding с Binance и запись колонки в CSV.

Такое разделение (данные / среда / обучение / оценка / сигнал) упрощает сопровождение и описание проекта: каждый модуль отвечает за один уровень абстракции.

## 3.2. Данные (формат, funding), признаки и причинность, разбиение train/validation

*В этот раздел вошли бывшие пункты: **3.3** «Данные», **3.4** «Признаки», **3.7** «Разбиение данных».*

### Формат CSV и загрузка

Функция `load_ohlcv_csv` требует столбцы **`open`, `high`, `low`, `close`, `volume`** (регистр в файле не важен — имена приводятся к нижнему регистру). Опционально задаётся **`funding_rate`** или **`funding`** — массив той же длины, что и число строк; пропуски заменяются нулём.

### Зачем колонка `open_time`

Для скрипта слияния с Binance каждой свече нужно однозначное время открытия в миллисекундах, чтобы сопоставить свечу с последней известной на этот момент ставкой funding (см. `merge_binance_funding.py`: для каждой свечи берётся последняя ставка с `fundingTime <= open_time`).

### Масштабирование funding на часовых свечах

На Binance funding начисляется дискретно (типично раз в восемь часов), а ставка в API относится к **полному интервалу** между funding-событиями. Если подставлять её **на каждую часовую свечу** без поправки, суммарное списание завысится примерно в восемь раз. В документации скрипта предлагается грубый коэффициент **`--funding-scale 0.125`** (1/8) или переход на свечи того же масштаба, что и начисление. В тексте работы это важно описать как **методологическое допущение** при агрегировании funding на таймфрейм.

### Синтетические данные

`make_sample_csv.py` генерирует 5000 свечей с логнормальным объёмом и случайными относительными движениями цены, записывает файл в **`data/sample_ohlcv.csv`**. Это позволяет проверить установку зависимостей и обучение до подключения реальных выгрузок.

### Признаки и принцип причинности

Функция `build_features` строит для каждого момента времени `t` вектор из трёх компонент (каждая — **z-нормализация по скользящему окну длины `window` только по прошлым данным**):

- лог-доходность закрытия относительно предыдущего закрытия;
- объём, выведенный в z-score по скользящему среднему и стандартному отклонению;
- относительный размах `(high − low) / close`, также в z-score по окну.

Маска **`valid`** помечает шаги, где окно полностью лежит в прошлом и все значения конечны; первые `window` индексов считаются невалидными.

**Почему так.** Обучение с подкреплением на финансовых рядах легко «сломать» утечкой будущего: если в наблюдение попадут статистики, вычисленные с участием точки `t` или правее, стратегия станет недостижима в реальном времени. Скользящие средние и дисперсии здесь считаются так, что на шаге `t` используются только цены и объёмы с индексов `≤ t−1` в духе rolling в pandas для признака в момент `t` (конкретику при проверке сверяют с кодом `utils/data.py`).

Дополнительно признаки ограничиваются по модулю (клиппинг), а `NaN`/`Inf` заменяются, чтобы нейросеть и среда не получали нечисловые состояния.

### Разбиение данных на обучение и валидацию

Доля **`train_ratio`** (по умолчанию 0.7) задаёт границу split по **числу строк** временного ряда. Граница дополнительно ограничивается снизу/сверху, чтобы слева и справа оставалось достаточно точек для окна и шага цены. Train-среда использует интервал **`[window, split)`** в смысле кода (старт эпизода выбирается случайно внутри train-диапазона), eval-среда — **`[split, n−2]`**.

**Почему так, а не k-fold по времени.** Для демонстрации полного конвейера достаточно одного временного разреза; при расширении исследования можно описать walk-forward как направление развития.

## 3.3. Среда обучения: постановка MDP (действие, наблюдение, переход, награда, эпизод)

*В этот раздел целиком вошёл бывший пункт **3.5** «Среда обучения: постановка MDP» со всеми подпунктами.*

### Действие

Непрерывное действие — **скаляр `w`**, целевая доля капитала в базовом активе (в упрощённой постановке «вся маржа в одной позиции»):

- режим **только long** (по умолчанию): `w ∈ [0, 1]` (0 — в кэше/вне позиции, 1 — полностью в активе);
- режим **long/short** (флаг `--short`): `w ∈ [−1, 1]` (−1 — выраженный шорт, 0 — нейтраль, +1 — лонг).

Политика PPO в SB3 для `MlpPolicy` выдаёт вещественное значение; среда обрезает его в допустимый интервал. Опционально включается ограничение **`max_abs_delta_w`**: за один дискретный шаг времени изменение `|w_new − w_old|` не может превышать заданного порога. Это моделирует **инерцию исполнения** и снижает переторговку, что ближе к реальному поэтапному набору позиции и стабилизирует обучение.

### Наблюдение

Размерность наблюдения: **`window × F + 2`**, где `F = 3` — число признаковых каналов на один временной слой. Вектор формируется как:

- «сплющенное» окно матрицы признаков `[t−window, …, t−1]` (в коде окно заканчивается на `t`, что соответствует использованию информации до текущего шага — согласовано с `step` и индексацией цены);
- дополнительно добавляются **`w`** (текущая позиция после клиппинга) и **`log(balance)`** — чтобы агент «видел» масштаб капитала и текущую экспозицию.

Наблюдение проходит `nan_to_num` с клиппингом, аналогично методу среды.

### Динамика баланса и доходность

Пусть `r_t = P_{t+1} / P_t` — простая гросс-доходность актива за шаг. Множитель портфеля за шаг (унифицированно для long и short):

**`growth_port = 1 + w_new · (r_t − 1)`**.

Комиссия моделируется как **доля от оборота позиции**: `fee = commission · |w_new − w_old|`, затем **`growth = growth_port · (1 − fee)`** (упрощение: издержки не разделяются на maker/taker и не зависят от объёма сделки в базовой валюте отдельно).

**Funding.** Если задан массив ставок по шагам или константа `funding_per_step`, баланс дополнительно умножается на **`1 − funding · w_new`** (при положительной ставке лонг платит, шорт получает — в терминах знака `w`).

Баланс обновляется мультипликативно; награда за шаг — **логарифм относительного прироста баланса** плюс (точнее минус) штраф за оборот:

**`reward = log(B_{t+1}/B_t) − turnover_penalty · |w_new − w_old|`**,

с клиппингом награды в разумные границы для численной устойчивости.

**Смысл награды.** Лог-прирост баланса за шаг согласован с аддитивностью по времени при перемножении гросс-множителей; штраф за оборот явно дисциплинирует частую смену веса и отражает комиссионные и прочие издержки активной торговли в агрегированном виде.

### Эпизод, границы индексов, reset

Индексы цены используются так, чтобы на шаге был доступен следующий `close`. Для обучения задаются **`start_idx` / `end_idx`**: на train-сегменте эпизод начинается со случайной допустимой точки (или с фиксированной при принудительных границах), на eval — фиксированный отрезок. Флаг **`max_episode_steps`** в тренировочном скрипте передаётся как `None`, поэтому длина эпизода определяется длиной сегмента данных (на валидации это даёт плоскую кривую «средней длины эпизода» в TensorBoard).

## 3.4. Алгоритм PPO и библиотека SB3, оценка, инференс, ограничения и выводы

*В этот раздел вошли бывшие пункты: **3.6** «PPO и SB3», **3.8** «eval_ppo», **3.9** «live_signal», **3.11** «Ограничения и выводы».*

### Выбор алгоритма: PPO и библиотека Stable-Baselines3

**PPO** выбран как стандарт для непрерывных действий: стабильные обновления политики, хорошая поддержка в SB3, предсказуемое потребление памяти по сравнению с некоторыми off-policy методами на длинных эпизодах.

В `train_ppo.py` задаются типичные гиперпараметры: политика **`MlpPolicy`**, `learning_rate=1e-4`, `n_steps=2048`, `batch_size=256`, `n_epochs=10`, `gamma=0.99`, `gae_lambda=0.95`, `clip_range=0.2`, `ent_coef=0.005`, `vf_coef=0.5`, `max_grad_norm=0.5`, `ortho_init=True`. Их удобно вынести в таблицу с кратким комментарием (скорость обучения, исследование за счёт энтропийного бонуса, стабилизация градиента).

**EvalCallback** периодически оценивает политику на **отложенном** сегменте и сохраняет лучший чекпоинт в каталог **`best/best_model.zip`** (важно не путать с путём без подпапки `best`). Финальные веса после полного числа шагов сохраняются как **`ppo_final.zip`**.

При включении **`--tensorboard`** логи пишутся в подкаталог **`tb`** внутри каталога сохранения рана; просмотр: `py -3 -m tensorboard.main --logdir <путь>`.

### Скрипт оценки `eval_ppo.py`

Скрипт загружает ту же CSV и (по возможности) ту же модель, строит среду на сегменте **`val`** или **`train`**, вызывает **`evaluate_policy`** на нескольких эпизодах (по умолчанию 3), печатает среднюю суммарную награду SB3, считает **buy-and-hold** на том же отрезке (одна покупка в начале сегмента с учётом комиссии входа) и дополнительно делает один детерминированный проход с подсчётом итогового баланса, суммарного оборота `|Δw|` и лог-доходности.

**Зачем дублирование с SB3.** `evaluate_policy` даёт сопоставимую с литературой метрику; ручной проход — прозрачные **баланс и оборот**, которые удобно привести в той же главе 3 при изложении результатов эксперимента.

Критично: флаги **`--short`**, **`--max-delta-w`**, **`--funding-per-step`**, **`--turnover-penalty`**, **`--commission`** должны совпадать с обучением, иначе сравнение некорректно.

### Инференс и `live_signal.py`

Для демонстрации «работы в реальном времени» без API-ключей скрипт:

- либо запрашивает последние свечи с публичного **`GET /fapi/v1/klines`** Binance USDT-M (без тяжёлого `exchangeInfo`), с повторами и увеличенным таймаутом;
- либо читает хвост ряда из локального **`--from-csv`** (удобно при нестабильной сети или для иллюстраций);

Далее строится то же наблюдение, что в среде (через **`utils/inference.py`**: `build_observation_vector`, `apply_max_delta`), загружается **`PPO.load`**, печатается сырое действие политики и вес после ограничения **`max_delta_w`**. **Ордера не отправляются** — это осознанное ограничение ответственности и объёма работ.

### Ограничения, риски и итоговые выводы (в рамках главы 3)

1. **Модель рынка упрощена:** один актив, непрерывный вес, комиссия от \|Δw\|, funding в линейной форме от `w`.
2. **Прибыльность на истории не гарантирует** результат в будущем; на коротких синтетических рядах возможны переобучение и случайные артефакты.
3. **Сравнение с buy-and-hold** на одном вал-сегменте информативно, но не исчерпывает сравнение стратегий; для серьёзного исследования приводят несколько активов, горизонтов и робастность к параметрам.
4. **Сеть и биржа:** публичные эндпоинты могут быть недоступны; удобно показывать **`--from-csv`** и заранее сохранённые скриншоты TensorBoard.

Раздел 3.4 логично завершить перечислением ограничений модели и краткими **выводами по работе**: какие команды запускались, какие метрики фиксировались (награда, баланс, оборот, сравнение с эталоном), какие гипотезы проверялись (влияние `turnover_penalty`, `max_delta_w`, funding). В принятой структуре документа глава 3 является **заключительной**; отдельной главы «Заключение» после неё не предполагается.
